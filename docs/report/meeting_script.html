Meeting Script — what to say

Defect Inspection Digital Twin · ~8–10 minute walkthrough · pairs with the slide deck
Before you start: have the GitHub repo open, and if you can, the Streamlit dashboard running (streamlit run stage2_decision/dashboard.py) for a live moment. Speak slowly; the script below is a guide, not a word-for-word read.

1. Opening — the problem ~1 min

"In circular manufacturing the goal is to keep parts and materials in use instead of throwing them away. But the hard part is the decision: when a used part comes back, should we reuse it, repair it, or recycle it? Today that's done manually — slow and subjective. I built a system that automates exactly that decision, end to end, in simulation."
  • Keep it concrete: "a worn part comes in, the system decides what to do with it."
  • Land the three words early: reuse / repair / recycle.

2. The big picture ~1 min

Show the pipeline diagram slide.
"It's one pipeline with four parts. A camera gives an image. A computer-vision model finds the defects. A grading step turns those defects into a decision with a confidence. And a 'digital twin' keeps a live record of every part we've seen. I built it in three layers so each one runs on its own."
  • Point at each box as you name it.
  • "Three layers" = Stage 1 vision, Stage 2 decision, Stage 3 robotics.

3. Stage 1 — seeing the defect ~2 min

Show the overlay slide (original | heatmap | overlay).
"Stage 1 is the computer vision. I used a pretrained PyTorch model — a ResNet — and fine-tuned it on surface-defect images. For each part it gives three things: the defect type, a heatmap showing where the defect is, and a confidence. The heatmap on the right is from a method called Grad-CAM — you can see it lights up exactly on the crack."
  • If asked why not draw exact outlines: "Grad-CAM only needs simple labels, not pixel-by-pixel annotation — much cheaper, and enough to locate the defect."
  • Mention the number: "about 88% accuracy on the demo set, trained in a minute."

4. Stage 2 — making the decision ~2 min

Show the grading-bands slide, then the decision-distribution slide.
"Stage 2 turns the defect into a decision. I compute a condition score from 0 to 100 from the defect type and how much of the surface it covers. A crack is structural, so it scores low and gets RECYCLE; a light scratch is cosmetic, scores in the middle, and gets REPAIR; a clean part scores high and gets REUSE. I kept this as a clear rule on purpose, so every decision can be explained and audited — not a black box."
"Every part is then saved to the digital twin with its score, decision and timestamp, and this dashboard shows the running statistics — including a 'recovery rate', the share of parts we keep in service."
If live: open the dashboard, pick a sample, show the decision badge update.

5. Stage 3 — the robotics layer ~1.5 min

Show the ROS 2 node-graph slide.
"Stage 3 wraps it in ROS 2, the standard robotics framework. Each job is its own node — camera, inspection, decision, digital twin — passing messages down the line. There's a small Gazebo scene of the inspection cell, a table with a camera over it, and RViz shows the live result with a colored marker: green for reuse, amber for repair, red for recycle."
  • Be honest: "ROS and Gazebo run on Linux, so I package it in Docker; I verified the structure here and it runs end-to-end in the container."
  • Emphasize reuse: "the nodes use the exact same model and grader — nothing is duplicated."

6. Why these choices ~1 min

"Every tool was chosen to be lightweight, free, and reproducible. Pretrained models so training is fast. Grad-CAM so I don't need expensive labels. A transparent grading rule so decisions are explainable. ROS 2 because it's the industry standard and keeps each part swappable. No paid services anywhere."

7. Closing ~30 sec

"So that's a complete, runnable demonstration that vision and ML can automate the reuse-repair-recycle decision for circular manufacturing — built in three independent layers, fully reproducible, and all open source. Happy to dive into any stage."

Likely questions & answers

Q: Is the accuracy real? "The 88% is on a synthetic demo set that proves the pipeline works offline. For real numbers I'd train on NEU-DET or the casting dataset — the code already supports both."
Q: Why a rule for grading, not machine learning? "Inspection decisions have to be auditable. A rule is explainable and tunable. I can add a learned grader later once I have labelled condition data, and compare."
Q: Does it run on real hardware? "It's simulation-only by design — no hardware needed. The ROS 2 structure is exactly what a real cell would use, so moving to a real camera and arm later is a small step."
Q: Why Grad-CAM instead of segmentation? "Segmentation needs pixel masks the public datasets don't have. Grad-CAM needs only image labels and still locates the defect. The interface is the same, so I can upgrade later."
Q: How long did the model take to train? "About a minute on CPU for the demo; a few minutes on a GPU for a real dataset — it's deliberately small and fast."
Timing: the whole thing is ~8–10 minutes. If you're short on time, cut Stage 3 detail and the live demo; keep the problem, Stage 1 overlay, and the decision bands — those three land the story.
